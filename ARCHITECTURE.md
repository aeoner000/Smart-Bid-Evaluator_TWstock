## 1. 總覽與設計哲學

本文件旨在闡述「台股競拍價格預測系統」的內部技術架構、資料處理流程與設計決策。我們的核心設計哲學是建立一個**高內聚、低耦合**的模組化系統。每個模組（爬蟲、清理、訓練、預測）都應能獨立運作與測試，同時透過清晰的介面（資料庫中的表）進行協作。此設計旨在最大化系統的**可維護性**、**可擴展性**與**可重現性**。

## 2. 資料清理管道 (Data Cleaning Pipeline)

資料清理是模型成功的基石。此管道的唯一職責是將從多個來源爬取、結構各異的原始資料（Raw Data），轉化為一個乾淨、寬格式、特徵豐富且可直接用於模型訓練的「單一事實表」(`all_features`)。

此流程由 `src/processors/feature_engineer.py` 模組負責，其設計遵循了經典的 ETL（擷取、轉換、載入）模式，但在實踐中，我們採用了更現代的「ELT-in-code」方法。

### 2.1 資料流轉圖
這張圖揭示了資料從原始、零散的狀態，一步步被加工成精煉、可用於模型的標準化資料集的完整過程。

整個流程是這樣運作的：

1.  **啟動與資料讀取：**
    *   流程的起點是 `FeatureEngineer` (特徵工程師) 模組。
    *   它首先會向我們的資料庫 (BigQuery) 發出請求，讀取所有在爬蟲階段收集到的「原始資料表」。這包括了核心的 `bid_info` (競拍基本資訊)，以及如 `history_price` (歷史股價)、`fin_stmts` (財務報表) 等十幾張各自獨立的附屬資訊表。

2.  **記憶體內資料合併 (`_combine_features_in_pandas`):**
    *   與其讓資料庫執行多重且複雜的 JOIN 操作，我們選擇將所有原始表讀取到程式的記憶體中，使用 Pandas 的 `merge` 功能進行合併。
    *   **處理流程：** 以 `bid_info` 為主表，將其他特徵表透過「證券代號」和「投標開始日」作為共同鑰匙，以 `left join` 的方式一一合併進來。
    *   **結果：** 這個步驟會產出一個非常寬的 `DataFrame`，它整合了單一競拍案件所有維度的原始特徵。

3.  **鏈式資料清理與特徵擴增 (`.pipe()`):**
    *   上一步產出的寬表會被送入一個由 `.pipe()` 串接起來的標準化處理流水線。資料會依序通過以下函數處理：
        *   `set_type`: 確保所有欄位的資料型態正確（例如，日期轉為日期格式，數字轉為數值格式）。
        *   `fill_nan` / `handle_missing_data`: 根據預設規則填補缺失值。例如，財務數據的空值可能補 0，而股價的空值可能用前一天的價格填充。
        *   `add_is_miss`: 為那些原本有缺失的欄位，額外新增一個「是否缺失」的標記欄位。這能讓模型學習到「缺失」本身可能帶有的資訊。
        *   `add_new_feature`: 根據領域知識，創造新的衍生特徵，例如計算 20 日均線、本益比、月報酬率等，以增強模型的預測能力。
    *   **結果：** 經過這條流水線，我們得到一個乾淨、格式統一且特徵更豐富的 `all_features` 表，並將它存回資料庫作為一個重要的「檢查點」。

4.  **偏態轉換與特徵選擇：**
    *   接著，系統會讀取 `all_features` 表，並為**每一個我們想要預測的目標**（例如「承銷價」或「上市後五日均價」）獨立進行以下操作：
        *   **`SkewTransformer` (偏態轉換器):** 許多金融數據分佈不均。此工具會學習如何將數據轉換成更接近常態分佈的樣態，讓模型更容易學習。這個轉換方法會被記錄下來。
        *   **`FeatureSelector` (特徵選擇器):** 從數百個特徵中，自動化地為當前的預測目標，挑選出最具預測力的 30 個特徵。這個「最佳特徵組合」的清單也會被記錄下來。

5.  **產出最終模型輸入表與轉換器物件：**
    *   **`CleanDB` (清理後資料表):** 系統會使用上一步選出的「最佳特徵組合」和「偏態轉換方法」，對應用的數據（訓練、測試、待預測集）進行處理，最終產出名為 `Train_*`, `Test_*`, `Predict_*` 的輕量化資料表。這些表只包含模型需要的欄位，可直接用於訓練。
    *   **`Artifacts` (轉換器物件):** 同時，上一步驟中學到的「偏態轉換方法」和「最佳特徵組合清單」，會被序列化成 `.joblib` 檔案並儲存起來。這確保了未來在預測新資料時，能執行與訓練時「完全一致」的資料轉換，避免模型因資料處理不一致而出錯。


### 2.2 清理邏輯與衍生特徵詳解

我們的清理流程是透過一系列 `pipe()` 鏈式調用實現的，這使得流程清晰且易於增減步驟。

- **資料合併 (`_combine_features_in_pandas`)**
    - **做了什麼:** 將 `bid_info` 主表與其他十幾個特徵表（如 `fin_stmts`, `history_price`）在 Pandas 中進行 `left_merge`。
    - **設計意圖:** 相較於在資料庫中執行複雜的 `JOIN`，將此邏輯移至應用層有三大優勢：
        1.  **降低資料庫負載：** 將計算壓力從 BigQuery 轉移到執行環境。
        2.  **靈活性與可讀性：** Python/Pandas 的合併邏輯比複雜的 SQL 更易於編寫、除錯和維護。
        3.  **效能：** 對於中等規模資料量（數十萬行內），一次性讀取後在記憶體中處理通常比多次網路往返的資料庫 `JOIN` 更快。

- **缺失值處理 (`fill_nan`, `add_is_miss`)**
    - **做了什麼:** 首先，根據預定義的策略（例如，財務欄位用 0 填充，價格相關用前值 `ffill` 填充）進行初步補值。接著，透過 `add_is_miss` 為某些關鍵欄位新增一個布林型態的衍生特徵，例如 `is_revenue_missing`。
    - **設計意圖:** 直接補值會丟失「資料是否缺失」這一重要資訊。某些情況下，資料的缺失本身就是一個強預測信號（例如，一家公司未能按時發布財報）。`is_miss` 特徵將此資訊明確地提供給模型。

- **衍生特徵工程 (`add_new_feature`)**
    - **做了什麼:** 根據業務理解創造新的特徵。例如：
        - **移動平均線 (Moving Averages):** `股價_20日均線`
        - **價格動量 (Momentum):** `月報酬率`
        - **財務比率 (Financial Ratios):** `本益比` (P/E Ratio), `股東權益報酬率` (ROE)
    - **設計意圖:** 原始特徵往往不足以捕捉市場動態。衍生特徵能將領域知識（Domain Knowledge）編碼到資料中，幫助模型學習更複雜的非線性關係。

- **偏態資料轉換 (`SkewTransformer`)**
    - **做了什麼:** 針對數值型特徵（包括 X 和 Y）進行 Box-Cox 或 Yeo-Johnson 轉換，使其分佈更接近常態分佈。
    - **設計意圖:** 金融資料（如成交量、價格）普遍呈現右偏分佈。許多機器學習模型（特別是線性模型和距離基礎模型）假設特徵為常態分佈。對資料進行脫偏處理可以提高這些模型的穩定性和預測能力。**關鍵設計：**我們為每個 Y 變數獨立儲存其對應的轉換器，並在預測後使用 `inverse_transform` 將結果還原回原始尺度。

- **特徵選擇 (`FeatureSelector`)**
    - **做了什麼:** 針對每個目標變數 Y，獨立地從數百個候選特徵中選出最具預測力的前 30 個特徵。
    - **設計意圖:** 這是一個關鍵的架構決策。我們**不**為所有模型使用同一組通用特徵。相反，我們認為預測「承銷價」和預測「市價」所需的最佳特徵組合是不同的。這種「客製化」的特徵選擇流程，旨在為每個預測目標最大化信噪比，降低模型複雜度，並有效防止過擬合。

### 2.3 資料庫 Schema

- **原始表 (Raw Tables):** 結構各異，例如 `bid_info`, `revenue_info`, `price_info` 等。它們是爬蟲的直接輸出，保留了原始的時間戳和結構。
- **中間表 (`all_features`):** 這是一個寬表（Wide Table），作為特徵工程的「檢查點」。它包含了所有公司的所有潛在特徵，`status` 欄位標記了其資料完整性。這是我們建模的「單一事實表」。
- **模型輸入表 (`Train_*`, `Test_*`, `Predict_*`):** 這些是最終餵給模型的表。它們的欄位是動態的，僅包含對應目標 Y 經過特徵選擇後的 30 個特徵，以及經過偏態轉換的目標變數本身。

## 3. 模組化設計與擴展性

`main.py` 的設計是系統解耦的核心。

- **職責分離:** `main.py` 本身不包含任何業務邏輯。它像一個指揮官，透過命令列參數（`crawl`, `process`, `train`）來調度不同的模組（工人）。
- **介面導向:** 模組之間不直接呼叫彼此的函式。它們唯一的溝通方式是透過 BigQuery 中的資料表。例如，`run_crawling_stage` 的輸出（新抓取的資料）是 `run_data_processing_stage` 的輸入。

**這種設計如何提升擴展性？**

1.  **水平擴展:** 每個階段都可以被容器化（例如 Docker）並獨立部署。當爬蟲任務繁重時，我們可以只增加爬蟲節點的數量，而無需改動訓練或預測服務。
2.  **非同步執行:** 可以將 `main.py` 的階段調度整合到 Airflow 或 Argo Workflows 等工作流引擎中。例如，設定一個每天凌晨執行 `crawl` 和 `process` 的 DAG，並在資料滿足特定條件時（例如，新資料累積超過 20 筆）才觸發 `train` 任務。
3.  **易於替換:** 如果我們未來決定用更高效的 Rust 爬蟲取代 Python 爬蟲，只要新的爬蟲能將結果以相同的 Schema 寫入 BigQuery，系統的其他部分完全無需變動。

## 4. 設計決策與效能考量

- **向量化運算:** 我們重度依賴 Pandas 和 NumPy 的向量化操作，避免在資料處理中使用 Python 的 `for` 迴圈。`.pipe()` 的鏈式風格確保了所有操作都是在整個 `DataFrame` 或 `Series` 上進行的，這是高效能的關鍵。
- **狀態與轉換器分離:** 將 `SkewTransformer` 和 `FeatureSelector` 這類「有狀態」的物件序列化儲存（`.joblib`）是一個核心的 MLOps 實踐。這確保了從訓練到預測，資料在每個環節都經過**完全相同**的轉換，從根本上杜絕了訓練-預測偏差（Training-Serving Skew）。
- **記憶體與資料庫的權衡:** 如前所述，我們選擇在應用層記憶體中進行資料合併。這是一個有意識的權衡。我們犧牲了單一節點的記憶體，換取了開發效率、靈活性以及對資料庫的低依賴。對於更大的資料集（億級行），此策略可能需要調整，例如轉向 Dask 或 Spark 進行分散式處理。

- **全域欄位命名標準化:**
    - **做了什麼:** 我們建立了一個全域的 `MASTER_RENAME_MAP` 字典，用於將原始爬蟲資料中的欄位名稱（如 `"競拍數量(張)"`）統一轉換為標準化格式（如 `"競拍數量_張"`）。這個過程會移除括號、百分比等特殊符號，並將單位直接整合進欄位名。
    - **設計意圖:** 此設計旨在從源頭解決資料不一致的問題，帶來多重好處：
        1.  **程式碼健壯性:** 標準化的欄位名是合法的 Python 變數名稱，可以直接透過 `df.欄位名` 的方式存取，避免了因特殊字元導致的語法錯誤。
        2.  **資料庫相容性:** 確保欄位名符合 SQL 標準，不會在存入資料庫時出錯。
        3.  **消除歧義:** 將單位（如 `_元`, `_張`, `_百分比`）直接寫入名稱，使得欄位的意義一目了然，降低了誤解資料的風險。
        4.  **集中管理:** 將所有重命名邏輯集中在一個映射表中，極大地方便了未來的維護。當來源欄位名變更時，只需修改此處即可。

- **爬蟲階段的資料回滾機制:**
    - **做了什麼:** 在 `main.py` 的爬蟲主流程中，我們圍繞著「特徵爬取站」設置了一個關鍵的 `try...except` 區塊。當此區塊內的任一爬蟲（如 `FinancialCrawler`）執行失敗時，系統會立即進入 `except` 流程，並啟動「資料回滾程序」。此程序會根據本次執行的任務索引 `pivotal_keys`，去刪除所有特徵爬蟲剛剛在資料庫中插入的對應資料。
    - **設計意圖:** 這是為了確保資料的「原子性」與「一致性」。一個競拍案件的特徵資料必須是「全部成功」或「全部失敗」的狀態，我們絕不允許資料庫中存在僅有部分特徵（例如只有股價，沒有財報）的「半成品」資料。此機制達成了以下目標：
        1.  **防止資料污染:** 避免不完整的資料流入下游的特徵工程與模型訓練階段，從而引發更難以追蹤的錯誤。
        2.  **簡化重試邏輯:** 當流程失敗時，由於髒資料已被清理，下一次執行時可以直接針對相同的任務進行重試，無需進行複雜的資料比對與修補。
        3.  **提升系統容錯力:** 正如程式碼所示，回滾程序自身也包含了錯誤處理（`if 'not found' in str(rollback_del_e).lower()`），即使在初次運行資料表尚不存在的情況下，系統也能優雅地處理，不會因此崩潰。

## 5. 核心流程概覽與架構強韌性分析

本節從宏觀角度審視整個台股競拍價格預測系統的端到端流程，並評估其架構的強韌性與潛在優化空間。

### 5.1 端到端資料流與模組互動

整個系統的運作，是透過一系列清晰定義的階段，將資料從原始狀態轉換為可預測結果。各階段之間高度解耦，主要透過 BigQuery 中的標準化資料表進行協作與溝通。以下為核心流程的逐步說明：

1.  **資料獲取階段 (Crawling Stage)**: 流程始於 `main.py` 啟動的爬蟲作業。`src/crawlers` 中的各類爬蟲會依序執行，從台灣證券交易所等外部來源抓取競拍相關的原始資料 (例如 `bid_info`, `fin_stmts`, `history_price` 等)。在抓取過程中，`finmind_manager.py` 會自動管理 API 憑證以規避限流問題。所有抓取到的資料都透過 `db_manager` (底層為 `bigquery_dao.py`) 以增量方式清洗並寫入 BigQuery 的原始資料表，確保資料的即時性與完整性。

2.  **資料處理與特徵工程階段 (Feature Engineering Stage)**: 完成資料獲取後，`main.py` 會觸發 `src/processors/feature_engineer.py` 模組。此模組從 BigQuery 讀取所有原始資料表，並在記憶體中進行一系列複雜的轉換：包括多源資料的合併、欄位清洗與類型統一、缺失值處理、以及根據領域知識生成豐富的衍生特徵。隨後，它會為每個特定的預測目標獨立執行特徵選擇，篩選出最具預測力的特徵集合。最終，這些經過標準化和偏度轉換處理後的訓練、測試和預測資料集 (`Train_*`, `Test_*`, `Predict_*` 表) 會被存回 BigQuery。同時，所有在特徵工程中學到的轉換器物件 (`SkewTransformer`, `FeatureSelector` 等) 也會被序列化並持久化，以確保預測時能應用完全一致的預處理邏輯。

3.  **模型訓練階段 (Model Training Stage)**: 接下來，`main.py` 進入模型訓練環節，啟動 `src/models/train_model/train.py`。該模組首先從 BigQuery 載入經過特徵工程的訓練與測試資料。然後，它會利用 `src/models/train_model/boost_automl.py` 模組，對多種增強型機器學習模型 (LGBM, XGBoost, CatBoost) 進行自動化的超參數優化和訓練。每個預測目標都會產生一個最佳模型，這個模型會與其評估分數和相關元數據一起被標記為「冠軍模型」，並透過 `storage_handler` 儲存，為後續的預測服務做好準備。

4.  **模型預測階段 (Model Prediction Stage)**: 最後，`main.py` 啟動 `src/models/train_model/predict.py` 模組執行預測任務。此模組會首先載入訓練階段所保存的冠軍模型、特徵列表和轉換器。隨後，它從 BigQuery 讀取最新的待預測資料 (`Predict_*` 表)，應用與訓練時完全一致的特徵預處理和轉換步驟。模型會對這些準備好的資料生成預測結果，並將最終的預測數值（經過反向轉換回到原始尺度）與原始的識別資訊 (`ID_COLS`) 合併，以格式統一的 `Result_*` 表儲存回 BigQuery，提供給下游應用或使用者進行決策參考。

### 5.2 架構強韌性與潛在優化點

在深入檢視核心模組後，我們識別出以下提升系統強韌性和可擴展性的關鍵點：

1.  **MLOps 實踐的深化**：當前系統已具備模型訓練與儲存的能力，但若要達到生產級 MLOps，需進一步導入模型註冊、版本控制、自動化部署 (例如模型服務 API)、持續監控與自動化再訓練等機制，以有效管理模型從開發到部署的整個生命週期。
2.  **資料管道的全面監控與告警**：目前系統具有日誌記錄，但缺乏集中的監控儀表板和自動告警機制。應整合雲端監控服務，對每個階段的執行狀態、資源使用、資料品質和模型性能進行實時監控，並在異常發生時觸發告警。
3.  **錯誤處理與容錯機制強化**：雖然各模組已有基礎錯誤處理，但在關鍵流程（如爬蟲的網路請求、BigQuery 的批次操作）中可加入更精細的重試邏輯和錯誤分類，以提升系統在面對外部不穩定因素時的韌性。
4.  **擴展性考量**：
    *   **資料量成長**：對於日益增長的資料量，`feature_engineer.py` 中的 Pandas 記憶體內操作可能會成為瓶頸。長遠來看，可能需要考慮部分資料處理任務轉移至分散式框架 (如 PySpark, Dask) 或 BigQuery 自身的 SQL 處理能力。
    *   **線上預測需求**：若未來有即時或低延遲預測的需求，則需要額外開發一個線上預測服務 (例如 REST API)，並可能需要引入 Feature Store 來統一線上和離線特徵。
5.  **配置與測試的優化**：
    *   **統一配置管理**: 將硬編碼的欄位名稱、閾值等配置項集中於 `config.yaml` 或環境變數中，以提高靈活性。
    *   **全面的測試覆蓋**: 擴展現有的單元測試，並增加整合測試和端到端測試，以確保各模組間協同工作的穩定性。
